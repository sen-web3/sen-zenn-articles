Soroiというステーブルコイン入金消込ツールを個人で開発しています。今回は自動照合ロジックの中身、特に「振込人名義が無い」という制約とどう向き合ったかについて書きます。

## 銀行明細との決定的な違い

銀行の入金消込ツールを触ったことがある方なら、振込人名義がどれほど強力な手がかりかご存知だと思います。「カ）タケシ」のような表記ゆれはあっても、名義さえあれば取引先の特定はほぼ終わったようなものです。

一方でオンチェーン送金にはそれがありません。送金元は0x始まりのアドレスだけで、そこに「誰が」の情報は乗っていません。取引先が事前にアドレスを教えてくれていれば別ですが、初回入金や新しいウォレットからの送金では手がかりがゼロの状態から突合を始めることになります。

Soroiで使える手がかりは結局のところ、金額・タイミング・（分かっていれば）アドレスの3つだけです。この記事ではこの3つをどう組み合わせて、誤マッチを抑えつつ実用的な自動化率を出すかという設計判断を書きます。

## 通貨違いを先に弾く

最初にやるべきはマッチング以前の話で、通貨が違うものは決して紐付けないというガードです。`lib/matching.ts` の `currenciesMatch` がそれに当たります。

```typescript
export function currenciesMatch(
  invoiceCurrency: Currency,
  transactionTokenSymbol: Currency
): boolean {
  return invoiceCurrency === transactionTokenSymbol;
}
```

一見当たり前に見えますが、この関数を独立させて共有したのには理由があります。自動照合（`matchInvoices`）と手動照合（`lib/actions.ts` の `createManualMatch`）の両方に通貨チェックが必要なのに、監査の2巡目で手動照合側にこのガードが漏れているのが見つかりました。USDCの請求書にUSDT入金を強制的に紐付けられる穴です。ロジックを2箇所に書くと必ずどちらかが劣化するので、判定は1関数に集約して両方から呼ぶ形に直しました。純関数の設計は理解しやすさのためだけでなく、こういう重複バグを機械的に潰すためでもあります。

## 金額とタイミングだけでどう当てるか

コア照合ルールは `MatchRule` の2種類、`exact_amount_within_window` と `amount_tolerance_within_window` です。名前の通り、金額が完全一致する候補を優先し、無ければ許容誤差内の候補まで広げます。

なぜ許容誤差が必要かというと、ステーブルコイン送金でもガス代の扱いや送金元がスワップを挟んでいるケースで、請求金額とわずかにずれた金額が入ってくることがあるからです。ここで誤差を0にすると取りこぼしが増え、緩めすぎると別件の入金を誤って掴んでしまいます。実運用で見えてきたのは、許容誤差を広げるほど自動化率は上がるが誤マッチのリスクも同時に上がるという単純なトレードオフで、これを緩和するために confidence というスコアを持たせています。

confidenceは「会計上正しいかどうか」を示すものではなく、あくまで機械的な一致候補としての確からしさです。このモジュールの冒頭コメントにも明記していますが、税理士法52条との関係で、Soroiは仕訳や課税判断を一切提示しない設計にしており、confidenceの意味づけもUI文言もそこに合わせています。

## 既知アドレスでconfidenceを底上げする

金額とタイミングだけでは同じ日に近い金額の入金が複数あると競合します。ここで効いてくるのが取引先アドレス帳です。`counterparties` / `counterparty_addresses` テーブルに登録済みのアドレスからの入金であれば、それは独立した強い証拠になります。

```typescript
export const KNOWN_ADDRESS_BONUS = 0.15;

function addressBookKey(counterpartyName: string, address: string): string {
  return `${counterpartyName.trim()} ${address.trim().toLowerCase()}`;
}
```

設計判断として大事にしたのは、この加点を「ゲート条件」ではなく「上乗せ」にしたことです。金額(0.6)＋時期(0.4)という既存の配点はそのまま残し、既知アドレス一致時だけ+0.15して1.0でクランプします。ゲート条件として組み込むと、アドレス帳未登録の初回入金がすべて弾かれてしまい、Soroiを導入した初日から使い物にならなくなります。加点方式にしたことで、複数候補が競合したときに貪欲割り当てで既知アドレス側が優先されつつ、未知アドレスでも金額とタイミングだけで候補には残る、というバランスを取れています。

アドレス帳の索引はDBの読み出し順を尊重して先勝ちにしています。同一アドレスが複数の取引先に登録されているケースは実運用でもゼロではなく、後から登録した方で上書きされると表示が不安定になるため、意図的に先勝ちにしました。

## 自動確定させない「要確認」フラグ

confidenceが高くても、未知アドレスや類似アドレス（コピペミスで1文字違いなど）からの入金は自動確定させたくありません。`MatchResult` に `needsReview` と `reviewReason` を持たせているのはこのためです。脅威モデルの検討時に洗い出した「なりすまし入金」「アドレス取り違え」のシナリオに対応するもので、confidenceのしきい値だけで押し切らず、疑わしいケースは人間の目を挟む設計にしています。

## まとめ

振込人名義という強力な手がかりが無い前提で消込を組むと、金額・タイミング・アドレスという弱い手がかりをどう積み上げてconfidenceに落とし込むかが設計の中心になります。ゲートと加点を分けること、疑わしい候補は自動確定させず人間に戻すこと、ロジックの重複を作らないこと。地味な判断の積み重ねですが、オンチェーン入金消込を実用的なところまで持っていくには、この地味さこそが必要だと感じています。
